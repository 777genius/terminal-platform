# Terminal Platform v1 Release Candidate Summary

This document is the repository handoff artifact for the first public v1 candidate.
It is intentionally stable and honest about both included scope and remaining publication gates.

## Product

Embeddable terminal platform in Rust with a daemon-first transport, a native PTY runtime,
and capability-gated `tmux` and `Zellij` adapters for Electron and other hosts.

## Support matrix

- `macOS + Linux` - `Native + tmux + Zellij`
- `Windows` - `Native + Zellij`
- `tmux` remains Unix-only in docs, CI, and acceptance

## Included in the v1 candidate

- daemon-first local transport over Unix sockets and Windows named pipes
- native PTY runtime with sessions, tabs, panes, topology snapshots, screen snapshots, and screen deltas
- conservative `tmux` import and control support
- conservative rich `Zellij 0.44+` import and ordered mutation support
- Node and Electron package surface through `napi-rs`
- documented C ABI leaf for non-Node embedders
- hosted CI, fuzz baseline, readiness audit, and recorded manual acceptance workflow

## What this release is for

- IDE terminal surfaces
- Electron apps with real terminal sessions
- agent or workspace products that need Rust-owned terminal runtime truth
- host applications that want a reusable terminal runtime layer instead of a UI-bound widget

## Not promised in v1

- fake parity between Native, `tmux`, and `Zellij`
- Windows `tmux` support
- full `Zellij` floating-pane parity
- signing, notarization, or installer families

## Known degraded semantics

- legacy `Zellij 0.43.x` import remains explicit `MissingCapability`
- Windows native currently vendors `portable-pty` to pin `CreatePseudoConsole(..., 0, ...)` instead of relying on upstream cursor-inherit or Win32-input flag combinations that are not yet proven stable for this host contract
- imported Unix `Zellij` pager validation for plain `less` is manual/degraded because `dump-screen` is not a trustworthy alternate-screen proof source there, so hosted automated proof uses `vim` plus `fzf`
- imported Windows `Zellij` fullscreen `fzf` validation remains manual/degraded because the hosted imported-screen path is not yet a trustworthy proof source there, so automated proof currently uses `vim` plus `less`
- imported backends are capability-gated foreign adapters, not product truth
- backend-specific behavior is exposed through capability truth instead of hidden fallback behavior

## Release proof required before tagging

- hosted `ci` on the release commit must be green:
  - `unix-matrix`
  - `windows-v1`
  - `governance`
  - `fuzz-baseline`
- `cargo run -p xtask -- verify-v1-readiness --require-recorded-passes` must pass
- recorded manual pass artifacts must exist for:
  - Electron embed
  - Unix `tmux`
  - Windows `Native + Zellij`
- the release PR generated by `release-plz` must stay clean and reviewable
